### 🧑‍💻 PROMPT ДЛЯ РОЛИ РАЗРАБОТЧИКА (вставь первым сообщением ИИ)

```text
# РОЛЬ: Lead Python Developer (ai_reviewer)
Ты отвечаешь за архитектуру и реализацию CLI-утилиты `ai_reviewer`. Твоя задача — писать модульный, тестируемый, production-ready код без вендор-лока и магических зависимостей.

## Стек и ограничения
- Python 3.10+
- Зависимости: `openai` (совместим с OpenRouter/Ollama/vLLM), `pydantic`, `tomli` (или stdlib `tomllib` для 3.11+), `rich` (только для форматирования вывода)
- Zero vendor lock-in: все LLM-вызовы через стандартный OpenAI-совместимый клиент
- Асинхронность: `asyncio` везде, где есть I/O
- Конфигурация: строго по XDG Base Directory Spec
- Ошибки: явные, логирование в stderr, exit-коды стандартизированы

## Принципы разработки
1. Строго следуй ТЗ. Не изобретай велосипеды и не добавляй фичи без прямого указания.
2. Каждый этап (Phase) должен быть самодостаточным для компиляции и тестирования.
3. Не генерируй код будущих этапов. Используй заглушки (stubs) там, где требуется интерфейс из следующих фаз.
4. Безопасность: API-ключи никогда не попадают в логи/историю. PII/секреты фильтруются до отправки в LLM.
5. Документация: docstrings, типы (type hints), краткие комментарии только для non-obvious логики.

## Протокол взаимодействия
- Я выдаю ТЗ по частям (Phase 0..N)
- Ты реализуешь ТОЛЬКО текущую фазу
- После завершения выводишь:
  • Изменённые/созданные файлы
  • Команды для проверки (`python -m pytest ...`, `ai_reviewer --help` и т.д.)
  • Статус: `READY FOR REVIEW`
- Останавливаешься. Ждёшь следующего куска ТЗ или правки.
```

### 📐 ТЕХНИЧЕСКОЕ ЗАДАНИЕ (модульное, для поэтапной сдачи)

#### 🟦 Phase 0: Скелет проекта и системная установка
**Цель:** Базовая структура, упаковка, entry-point, минимальный CLI.
**Файлы:** `pyproject.toml`, `src/ai_reviewer/__init__.py`, `src/ai_reviewer/cli.py`
**Требования:**
- `pyproject.toml`: build-system `setuptools`, deps `openai>=1.30`, `pydantic>=2.0`. `[project.scripts] ai_reviewer = ai_reviewer.cli:main`
- `cli.py`: `argparse` с аргументами `--repo` (default `Path.cwd()`), `--mode`, `--dry-run`, `--verbose`
- При запуске без аргументов выводит справку и `exit(0)`
- Поддержка `pip install .` и `pipx install .`
**Тесты/валидация:** `python -m ai_reviewer.cli --help` работает, `pipx install .` создаёт команду `ai_reviewer`.

#### 🟩 Phase 1: Конфигурация и управление ключами
**Цель:** Иерархическая загрузка конфигов, безопасное хранение API-ключей.
**Файлы:** `src/ai_reviewer/config.py`, `src/ai_reviewer/models.py` (Pydantic)
**Требования:**
- Иерархия (перекрытие сверху вниз): CLI args → `./ai-reviewer.toml` → `~/.config/ai-reviewer/config.toml` → `~/.config/ai-reviewer/credentials.toml` → built-in defaults
- `credentials.toml`: `[auth] provider="openrouter"`, `api_key`, `base_url` (fallback `https://openrouter.ai/api/v1`)
- Поддержка переменных окружения: `AI_REVIEWER_API_KEY`, `OPENROUTER_API_KEY`, `AI_REVIEWER_MODEL`
- Валидация через Pydantic. Ошибка при отсутствии ключа: `exit(3)`
- Пути по XDG: `~/.config/ai-reviewer/`, `~/.local/share/ai-reviewer/`, `~/.cache/ai-reviewer/`
**Тесты:** `ai_reviewer --repo . --model qwen/qwen3.5-flash` корректно резолвит конфиг. При отсутствии ключа → понятная ошибка в stderr.

#### 🟨 Phase 2: Резолвинг входных данных
**Цель:** Гибкий парсинг целей ревью, поддержка пайпов, git, PR.
**Файлы:** `src/ai_reviewer/target_resolver.py`
**Требования:**
- Поддержка аргументов: путь к файлу, директория, glob (`"*.py"`), stdin (если `sys.stdin.isatty()==False`)
- Git-режим: `--diff-only` → `git diff --cached` или `git diff origin/main...HEAD`
- PR-режим: `--pr <url_or_number>` → валидация URL, извлечение `repo`, `pr_number` (пока только парсинг, без API-вызовов)
- Возвращает объект `ReviewTarget` со списком файлов, типом источника, хэшем содержимого
- Ошибка при недоступных путях/битом git → `exit(1)`
**Тесты:** `echo "print('x')" | ai_reviewer --mode light` читает stdin. `ai_reviewer "src/**/*.py"` резолвит glob.

#### 🟧 Phase 3: LLM-клиент и учёт бюджета
**Цель:** Обёртка над OpenAI SDK, оценка токенов, обработка ошибок API.
**Файлы:** `src/ai_reviewer/llm_client.py`, `src/ai_reviewer/token_tracker.py`
**Требования:**
- Async-клиент на базе `openai.AsyncOpenAI` с `base_url` из конфига
- Метод `chat_completion(messages, model, max_tokens, budget_limit)` с retry (exponential backoff, 3 попытки)
- `token_tracker`: считает `input`, `output`, `cached`, `cost_usd` на основе цен из конфига (или default map)
- При превышении `80%` бюджета → бросает `BudgetExceededError`
- Логирование метрик запроса (без тела промпта/ответа)
**Тесты:** Мок `openai` SDK. Проверка retry при 429/503. Проверка срабатывания лимита.

#### 🟥 Phase 4: Пайплайн ревью и режимы
**Цель:** Многоэтапный runner, промпты, fail-fast логика.
**Файлы:** `src/ai_reviewer/pipeline.py`, `src/ai_reviewer/prompts/` (j2 или raw strings)
**Требования:**
- Режимы: `light` (1 шаг), `standard` (2 шага), `deep` (3 шага + агрегация)
- Этапы: `PRECHECK` → `DIFF_ANALYSIS` → `CONTEXT_READ` → `VERIFY`
- Fail-fast: если `critical_findings >= threshold` (default 1) → остановка пайплайна, возврат `BLOCKED`
- Промпты вынесены в отдельные строки/файлы, принимают `target`, `diff`, `context`, `findings_history`
- Формат ответа LLM: строгий JSON `{"verdict": "PASS|BLOCK|WARN", "findings": [...]}`
- `response_format={"type": "json_object"}` включён по умолчанию
**Тесты:** Юнит-тесты на фейковых LLM-ответах. Проверка прерывания при `critical=1`. Валидация JSON-выхода.

#### 🟪 Phase 5: Pre-check линтеры и безопасность
**Цель:** Локальные проверки до LLM, фильтрация секретов/PII.
**Файлы:** `src/ai_reviewer/precheck.py`, `src/ai_reviewer/filters.py`
**Требования:**
- `precheck.py`: запуск `ruff check --select=E9,F63,F7,F82`, `bandit -ll`. Парсинг JSON-вывода.
- `filters.py`: Regex-скан на `AWS_KEY`, `PRIVATE_KEY`, `password=`. Опциональный `--sanitize` → замена `[REDACTED]`
- Интеграция в пайплайн: если `precheck` нашёт `SyntaxError` или секрет → `BLOCKED`, LLM не вызывается
- Кэширование результатов линтера по `sha256(file_content + tool_version)`
**Тесты:** Файл с `<<<<<<< HEAD` → блок. Файл с `sk-or-...` → блок после sanitize → пропуск.

#### 🟦 Phase 6: Вывод, история и CI/CD
**Цель:** Форматирование результатов, журналирование, exit-коды.
**Файлы:** `src/ai_reviewer/output.py`, `src/ai_reviewer/history.py`
**Требования:**
- `--format markdown` (человек), `--format json` (машина/CI)
- История: JSONL в `~/.local/share/ai-reviewer/history/`. Поля: `ts`, `mode`, `model`, `target_hash`, `tokens`, `cost`, `verdict`, `findings_count`
- Exit-коды: `0`=PASS, `1`=WARNINGS, `2`=BLOCKED, `3`=ERROR
- `--dry-run`: выводит план, оценку токенов/стоимости, список файлов, не вызывает API
- `--no-history`: отключает запись
**Тесты:** Проверка кодов возврата при разных вердиктах. Валидация JSONL структуры. Dry-run не шлёт запросы.

#### 🟨 Phase 7: Кэширование и хуки (финализация)
**Цель:** Инкрементальность, pre/post скрипты, полировка.
**Файлы:** `src/ai_reviewer/cache.py`, `src/ai_reviewer/hooks.py`
**Требования:**
- Кэш ответов LLM: `sha256(target + model + prompt_version + stage)`. Пропуск неизменённых файлов/этапов
- Хуки в `ai-reviewer.toml`: `[hooks] pre_process="...", post_process="..."`. Запуск через `subprocess`, передача пути к отчету через env `AI_REPORT_PATH`
- Graceful shutdown: Ctrl+C → сохранение partial-отчета, cleanup tmp
- `ai_reviewer history --last 5` → чтение JSONL, вывод таблицы
**Тесты:** Кэш работает при повторном запуске. Хук `post_process` получает отчет и возвращает 0. Ctrl+C корректно завершает процесс.
